## Executive Summary

**Feasibility: FULLY VIABLE** ✅

Binance Simple Earn Flexible API is production-ready with official endpoints, CCXT implicit method support possible, and no testnet limitations. Integration into rebalance-bot is technically straightforward. Flexible products offer 2-6% APY on USDT/stablecoins with instant redemption, making them practical for a rebalancing bot. Main trade-offs: wallet separation (must redeem before selling), daily redemption limits, and minor execution complexity.

## 1. API Availability & Endpoints

### Official Support: YES
Binance formally launched Simple Earn API in **Q3 2023**. Fully documented at [Binance Open Platform](https://developers.binance.com/docs/simple_earn/).

### Available Endpoints (Simple Earn Flexible)

**Core Operations** (all support HMAC-SHA256, RSA, Ed25519 signing):

| Endpoint | Method | Rate Limit | Purpose |
|----------|--------|-----------|---------|
| `/sapi/v1/simple-earn/flexible/list` | GET | 1/10s (IP) | List available flexible products + APY |
| `/sapi/v1/simple-earn/flexible/subscribe` | POST | 1/10s (IP) | Subscribe to flexible product |
| `/sapi/v1/simple-earn/flexible/redeem` | POST | 1/3s (IP) | Redeem from flexible product |
| `/sapi/v1/simple-earn/flexible/position` | GET | 1/10s (IP) | Get current positions & earned rewards |
| `/sapi/v1/simple-earn/flexible/history` | GET | 1/10s (IP) | Transaction history |
| `/sapi/v1/simple-earn/account` | GET | 1/10s (IP) | Summary of all earn positions |

**Query Endpoints** (read-only):
- `/sapi/v1/simple-earn/flexible/productList` — Get product APY details
- `/sapi/v1/simple-earn/flexible/quota` — Check personal subscription limits

### Rate Limits (Important)
- **IP-based**: 12,000 req/min per endpoint group (`/sapi/*`)
- **Account-based**: 180,000 req/min (with UID restriction enabled)
- Testnet subject to same limits; respect 1/3s per-account throttle on redemptions

### Authentication
All endpoints require `X-MBX-APIKEY` header + signature.

**Signing updates (effective ~2026-01-15)**:
Payloads must be percent-encoded before signature computation. Latest Binance SDKs handle this automatically.

## 2. CCXT Support Assessment

### Direct CCXT Methods: NOT YET

CCXT does not yet have **unified methods** for `fetchSavings()`, `createSavings()`, etc.

**Source**: [CCXT GitHub Issue](https://github.com/ccxt/ccxt) confirms Savings/Earn is on roadmap but not yet unified across exchanges. Rationale: Earn products are exchange-specific with different APY models.

### Implicit Method Pattern: YES ✅

CCXT supports **implicit API method generation** for Binance-specific endpoints.

**Pattern**: `[api_prefix][HTTP_Method][Endpoint_Parts](params)`

**Examples**:
```typescript
// Get flexible product list
const products = await exchange.sapiGetSimpleEarnFlexibleList();

// Subscribe to flexible earn
const subscription = await exchange.sapiPostSimpleEarnFlexibleSubscribe({
  productId: 'BNB001',
  amount: 10
});

// Redeem from flexible earn
const redemption = await exchange.sapiPostSimpleEarnFlexibleRedeem({
  productId: 'BNB001',
  amount: 5,
  destAccount: 'SPOT' // Optional: defaults to SPOT
});

// Get current position
const position = await exchange.sapiGetSimpleEarnFlexiblePosition({
  productId: 'BNB001'
});
```

**Source**: [CCXT Implicit Methods Documentation](https://dev.to/macanudo527/using-ccxt-implicit-api-methods-1on8) + Binance API swagger definitions on GitHub.

**Why this matters**: The method names are auto-generated by CCXT's `ImplicitAPI` class from Binance's endpoint schema. No manual CCXT PR needed; works today if Binance endpoint is in their Swagger spec.

### Verification Required
Test implicit method availability in your CCXT version:
```typescript
const exchange = new ccxt.binance({ apiKey: '...', secret: '...' });

// Check if implicit method exists
if (typeof exchange.sapiPostSimpleEarnFlexibleSubscribe === 'function') {
  console.log('✓ Implicit method available');
} else {
  console.warn('✗ Fall back to direct REST call');
}
```

### Fallback: Direct REST API
If implicit methods unavailable, call Binance REST API directly via CCXT's lower-level API:

```typescript
const response = await exchange.privatePost('simple-earn/flexible/subscribe', {
  productId: 'BNB001',
  amount: 10
});
```

Or use native fetch/axios if needed (less elegant but works).

## 5. How Flexible Earn Works (Operationally)

### Mechanics

**Subscribe** (allocate capital to earn):
1. Call `/sapi/v1/simple-earn/flexible/subscribe` with product ID + amount
2. Funds deducted from Spot wallet immediately
3. Appear in separate "Earn Wallet" (not spendable for trading)
4. Rewards accrue every **1 minute** for **Real-Time APR**
5. Bonus tier rewards distributed daily to Spot wallet

**Earn** (automatic):
- Rewards accrue without action; no reinvestment needed
- Real-time APR rewards accumulate on the deposit amount in Earn wallet
- Auto-subscribe feature available: daily reinvest earned rewards back into Earn

**Redeem** (withdraw from earn):
1. Call `/sapi/v1/simple-earn/flexible/redeem` with product ID + amount
2. Assets returned to Spot wallet **immediately** (under normal conditions)
3. Subject to **daily redemption limits** per product (varies; typically $1M+)
4. Potential delays: extreme volatility, network congestion, or mass redemption from other users → up to several hours

### Redemption Latency (Critical for Bot)

**Normal case**: Spot wallet receives redeemed funds within **seconds**.

**Degraded case**: Extreme market conditions, network load, or coordinated mass redemptions → **up to 4-6 hours** reported in user forums (rare).

**For rebalancing bot**: Assume **2-5 second latency** for normal operation. If rebalance needs to happen **immediately** (e.g., drift spike), must account for redemption time in order sequencing.

### Daily Redemption Limits

- Binance enforces daily redemption caps **per product per account**
- Not publicly disclosed exact limits; typically $1M+ for major products (USDT, USDC)
- Check via API: `/sapi/v1/simple-earn/flexible/history` → `redeemedAmount` fields to track daily usage
- Impact: Bot cannot redeem unlimited amounts in single rebalance if approaching daily cap

## 6. Portfolio Balance Visibility

### Key Architectural Impact

**Earn Wallet ≠ Spot Wallet**

- **Earn Wallet**: Separate balance. Assets in Earn cannot be directly traded.
- **Spot Wallet**: Trading balance. Earn assets must be redeemed to Spot before selling.

**Balance Queries**:

```typescript
// Fetch balance
const balances = await exchange.fetchBalance();

// Structure:
// balances = {
//   USDT: { free: 1000, used: 0, total: 1000 },  // Spot wallet only
//   BTC: { free: 0.5, used: 0, total: 0.5 },
//   ...
// }

// EARN BALANCE NOT INCLUDED IN SPOT BALANCE
```

**You must query Earn position separately**:
```typescript
const earnPosition = await exchange.sapiGetSimpleEarnFlexiblePosition({
  productId: 'USDT001'
});
// earnPosition = { productId: '...', asset: 'USDT', amount: 5000, ... }
```

**Impact on portfolio tracking**:
1. Portfolio dashboard must sum: `spot_balance + earn_balance` for true net position
2. Rebalancer cannot use Spot-only balance for allocation calculations
3. Test assumption: bot currently tracks only `fetchBalance()` (Spot) → must add Earn balance fetches

## 7. Integration Design: Recommended Architecture

### Integration Layer (New Module)

Create `src/exchange/simple-earn-manager.ts`:

```typescript
interface FlexibleEarnPosition {
  productId: string;
  asset: string;
  amount: number;
  rewardAsset: string;
  earnedAmount: number;
  APY: number;
  subscriptionDateTime: number;
}

interface SimpleEarnManager {
  // Get all active flexible earn positions
  getFlexiblePositions(): Promise<FlexibleEarnPosition[]>;

  // Subscribe to flexible earn
  subscribe(productId: string, amount: number): Promise<void>;

  // Redeem from flexible earn
  redeem(productId: string, amount: number, destAccount?: string): Promise<void>;

  // Get available products + rates
  getFlexibleProducts(): Promise<ProductInfo[]>;

  // Get subscription quota for product
  getQuota(productId: string): Promise<QuotaInfo>;
}
```

### Portfolio Manager Integration

Update `src/portfolio/portfolio-service.ts` to include Earn balances:

```typescript
interface PortfolioSnapshot {
  spotBalance: Record<string, number>;
  earnBalance: Record<string, number>;      // NEW
  totalBalance: Record<string, number>;     // Aggregate
  allocations: Allocation[];
  drift: number;
}

async getPortfolio(): Promise<PortfolioSnapshot> {
  const spotBal = await exchange.fetchBalance();
  const earnPos = await simpleEarnManager.getFlexiblePositions();
  
  // Aggregate into totalBalance
  return {
    spotBalance: spotBal,
    earnBalance: aggrEarnAssets(earnPos),
    totalBalance: merge(spotBal, earnBalance),
    allocations: computeAllocations(totalBalance),
    drift: computeDrift(allocations)
  };
}
```

### Rebalancer Workflow

**Current flow**: Detect drift → calculate target → execute trades

**With Earn integration**:

1. **Detect drift** (using total balance: Spot + Earn)
2. **Calculate redemptions** needed for sell-side (e.g., if selling BTC but BTC is in Earn)
3. **Stage redemptions** (async, wait for completion)
4. **Execute trades** (buy/sell)
5. **Subscribe remaining capital** (post-rebalance, optional)

```typescript
async executeRebalance(targetAllocations: Allocations) {
  // 1. Identify positions needing redemption
  const requiredRedemptions = identifyRedemptions(currentState, targetAllocations);

  // 2. Trigger redemptions (parallel)
  await Promise.all(requiredRedemptions.map(r => 
    simpleEarnManager.redeem(r.productId, r.amount)
  ));

  // 3. Poll Spot balance until redemptions settle (timeout: 30s)
  await waitForSettlement(requiredRedemptions, 30000);

  // 4. Execute trades
  await executeTradesInOrder(trades);

  // 5. Subscribe residual capital to Earn (optional)
  if (config.enableEarnAfterRebalance) {
    await subscribeResidue(trades.residualCapital);
  }
}
```

### DCA + Earn Integration

**Proposed workflow**:
1. DCA buy executes (buy BTC, USDT, etc. at scheduled time)
2. Immediately after fill: offer to subscribe DCA proceeds to Earn
3. Config option: `AUTO_EARN_AFTER_DCA=true` + `EARN_SUBSCRIBE_DELAY_HOURS=1`
4. Wait N hours, then auto-subscribe if bot still running (user can manually redeem before)

**Benefit**: DCA capital earning ~3-6% APY between rebalance cycles (typically weekly/monthly).

## 8. Risks & Mitigation

### Risk 1: Redemption Delays

**Scenario**: Rebalance needed, but assets locked in Earn with slow redemption.

**Mitigation**:
- Pre-calculate redemptions required for next rebalance cycle
- Redeem 12-24 hours before threshold expected to breach
- Maintain 10-20% "reserve" in Spot (not subscribed) for emergency rebalances
- Config: `EARN_SUBSCRIPTION_MAX_PERCENT = 0.8` (reserve 20% liquid)

### Risk 2: Daily Redemption Limits Hit

**Scenario**: Bot hits daily redemption cap mid-rebalance.

**Mitigation**:
- Query daily usage via `/sapi/v1/simple-earn/flexible/history` before rebalance
- If approaching limit, stage redemptions across multiple days
- Alert user (Telegram) if daily cap constrains rebalance timing
- Fallback: skip Earn step, trade from reduced Spot balance

### Risk 3: Earn Balance Not Included in Allocations

**Scenario**: Portfolio drift calculation using only Spot balance → incorrect targets.

**Mitigation**:
- Always aggregate: `total_balance = spot + earn`
- Dashboard displays both separately + combined
- Tests verify Earn balance fetching works

### Risk 4: APY Changes / Product Discontinuation

**Scenario**: Binance changes APY rates or stops offering product.

**Mitigation**:
- APY changes: Expected + acceptable (bot just earns different rate)
- Product discontinuation: Rare; Binance would force redemption with notice
- Graceful handling: If product not in list, skip subscription, but don't fail rebalance

### Risk 5: Smart Contract / Custody Risk

**Assessment**: Binance Earn uses **Binance's own custody** (not self-custody via smart contract).

- Assets held in **Binance wallet**, not protocol tokens
- Risk profile ≈ holding Spot balance (exchange counterparty risk)
- **No smart contract audits needed** (not DeFi protocol)
- **Implicit risk**: If Binance exchange compromised, Earn balances also at risk

**Mitigation**: Standard best practices apply (not Earn-specific)
- Use sub-account with limited API permissions
- Rotate API keys monthly
- Enable IP whitelisting
- Use hardware wallet for cold storage, bot only trades hot wallet

## 12. API Call Examples (Code-Ready)

### Get Available Products

```typescript
async function listFlexibleProducts(exchange: ccxt.binance) {
  try {
    const products = await exchange.sapiGetSimpleEarnFlexibleProductList();
    return products.rows.map(p => ({
      productId: p.productId,
      asset: p.asset,
      APY: parseFloat(p.apr) * 100,
      minAmount: parseFloat(p.minAmount)
    }));
  } catch (error) {
    console.error('Failed to fetch products:', error);
    throw error;
  }
}
```

### Subscribe to Product

```typescript
async function subscribeToEarn(
  exchange: ccxt.binance,
  productId: string,
  amount: number
) {
  try {
    const result = await exchange.sapiPostSimpleEarnFlexibleSubscribe({
      productId,
      amount: amount.toString()
    });
    return { transactionId: result.transactionId, timestamp: result.timestamp };
  } catch (error) {
    if (error.code === 'INSUFFICIENT_BALANCE') {
      throw new Error(`Insufficient balance to subscribe ${amount}`);
    }
    throw error;
  }
}
```

### Redeem from Earn

```typescript
async function redeemFromEarn(
  exchange: ccxt.binance,
  productId: string,
  amount: number
) {
  try {
    const result = await exchange.sapiPostSimpleEarnFlexibleRedeem({
      productId,
      amount: amount.toString(),
      destAccount: 'SPOT' // explicit destination
    });
    return { transactionId: result.transactionId };
  } catch (error) {
    if (error.code === 'EXCEED_DAILY_LIMIT') {
      throw new Error('Daily redemption limit exceeded');
    }
    throw error;
  }
}
```

### Get Current Positions

```typescript
async function getEarnBalance(exchange: ccxt.binance) {
  try {
    const account = await exchange.sapiGetSimpleEarnAccount();
    return {
      totalBalance: parseFloat(account.totalBalance), // aggregated
      totalFlexible: parseFloat(account.totalFlexibleBalance),
      totalLocked: parseFloat(account.totalLockedBalance),
      positions: account.rows.map(p => ({
        productId: p.productId,
        asset: p.asset,
        amount: parseFloat(p.amount),
        earned: parseFloat(p.earnedAmount)
      }))
    };
  } catch (error) {
    console.error('Failed to fetch earn balance:', error);
    throw error;
  }
}
```
